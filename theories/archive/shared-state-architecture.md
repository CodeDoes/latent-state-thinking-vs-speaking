# Shared-State Architecture Sketch

**Date:** 2026-07-16  
**Status:** Design phase, not yet implemented

## Core Architecture

Three coupled state vectors operating at different timescales:

1. **Byte-level state (RWKV):** fast, local, sequential
   - Processes bytes one at a time
   - Accumulates state via RWKV recurrence
   - Tracks "surprise" (entropy of next-byte prediction)
   - When surprised → sends state to patch-model as a "patch"

2. **Patch-model state:** slow, global, contextual
   - Receives byte-level-state as patches (triggered by surprise)
   - Processes patches sequentially
   - Outputs:
     - (a) New byte-level-state (correction/continuation)
     - (b) Predicted next patch(es) (lookahead)
   - The predicted patch is a "direction" — a vector pointing toward what the next patch should be
   - It encodes both content (what the word is) and context (where it appears, what's coming next)

3. **Decoder state:** generates bytes using all available context
   - Input: byte-level-state + current patch + predicted future patches
   - Generates bytes sequentially
   - Tracks uncertainty about next byte
   - When uncertain → signal "token complete" (dynamic boundary)

## Key Design Principles

### 1. Byte-level-state = Patch (No Separate Encoding)

The byte-level-state IS the patch. When byte-level-model is surprised, it dumps its accumulated state to patch-model. No separate encoding step, no mean-pooling, no cross-attention pooling.

**Similarity to BLT's "smear":** BLT uses hash n-gram embeddings + cross-attention pooling to create patch representations from byte sequences. The result is a "smeared" representation that integrates information from multiple bytes. Our approach is similar in spirit: the RWKV state has integrated information from bytes via recurrence. When we send it to patch-model, it's a compressed representation of "what these bytes mean in context."

**Difference from BLT:** BLT uses attention pooling over byte embeddings. We use RWKV state accumulation. Both achieve compression, but RWKV is more efficient (linear scaling) and maintains temporal order naturally.

### 2. Patch-Model Output = Direction

The patch-model doesn't just compress — it predicts. Its output is a "direction" vector that points toward what the next patch should be.

**What this encodes:**
- Content: what the next word/token is
- Context: where it appears in the sequence, what's coming next
- Trajectory: the "momentum" of the sequence

**Analogy:** This is like a "meta-token" that encodes both the token embedding and the contextual embedding. In transformer terms, it's like combining the token embedding with the position embedding and the attention output.

**Why this matters:** The decoder can use this "direction" as lookahead context. Even if byte-level-model doesn't need it yet, the decoder benefits from knowing what's coming next.

### 3. Dynamic Token Boundaries (Decoder Uncertainty)

Token boundaries are determined by decoder uncertainty, not fixed patch sizes or external entropy models.

**Mechanism:**
- Decoder generates bytes sequentially
- At each byte, decoder computes uncertainty (entropy of next-byte prediction)
- When uncertainty exceeds threshold → signal "token complete"
- This is dynamic and learned, not fixed or heuristic

**Why this works:** When the decoder is uncertain, it means "I've reached the end of a predictable unit." This naturally identifies token boundaries (word boundaries, phrase boundaries, etc.).

**Advantage over fixed patches:** Fixed patches waste capacity on predictable regions (common words) and don't allocate enough capacity to surprising regions (rare words, boundaries). Dynamic boundaries adapt to content complexity.

## Training

**Objective:** Next-byte prediction (cross-entropy)

**Loss components:**
1. Byte-level loss: byte-level-model predicts next byte
2. Patch-level loss: patch-model predicts next patch (MSE or cosine similarity)
3. Decoder loss: decoder predicts next byte using all context

**Training flow:**
1. Byte-level-model processes bytes, accumulates state
2. When surprised, send state to patch-model
3. Patch-model processes patch, outputs new state + predicted next patch
4. Decoder generates bytes using byte-level-state + current patch + predicted future patches
5. Compute loss on generated bytes
6. Backprop through everything

**Gradient flow:**
- Byte-level-model learns to accumulate useful state
- Patch-model learns to compress and predict patches
- Decoder learns to use all available context

## Implementation Challenges

### 1. Surprise Trigger

How does byte-level-model decide when it's "surprised"?

**Option A: Entropy threshold**
- Compute entropy of next-byte prediction: H = -Σ p(x) log p(x)
- When H > threshold → trigger patch boundary
- Threshold can be fixed or learned

**Option B: State norm**
- Track state vector norm (or some other measure of "fullness")
- When norm > threshold → trigger patch boundary
- This is capacity-based, not surprise-based

**Option C: Learned trigger**
- Byte-level-model outputs a scalar "boundary probability"
- Trained end-to-end
- More flexible but more complex

**Recommendation:** Start with Option A (entropy threshold), keep it simple.

### 2. Patch-Model Output

Patch-model outputs:
- (a) New byte-level-state
- (b) Predicted next patch(es)

**How to combine these?**

**Option A: Separate outputs**
- Patch-model has two output heads: state-head and patch-head
- State-head outputs new byte-level-state
- Patch-head outputs predicted next patch
- Both are trained with separate losses

**Option B: Single output, split**
- Patch-model outputs a single vector
- First half is new byte-level-state
- Second half is predicted next patch
- Simpler but less flexible

**Option C: Recurrence**
- Patch-model processes patches sequentially
- Output at time t is new byte-level-state
- Output at time t+1 is predicted next patch
- This requires patch-model to run for multiple steps

**Recommendation:** Start with Option A (separate outputs), clearest signal.

### 3. Decoder Lookahead

Decoder gets:
- Byte-level-state (current)
- Current patch (from patch-model)
- Predicted future patches (from patch-model)

**How to combine these?**

**Option A: Concatenation**
- Concatenate all vectors, project to decoder input dim
- Simple but might not capture interactions well

**Option B: Cross-attention**
- Decoder queries attend to patch-model outputs
- More flexible, captures interactions
- More expensive

**Option C: Gating**
- Use learned gates to combine byte-level-state and patch context
- Middle ground between concatenation and cross-attention

**Recommendation:** Start with Option A (concatenation), keep it simple. Add cross-attention later if needed.

### 4. Token Completion

Decoder signals "token complete" when uncertain.

**How to implement this?**

**Option A: Hard threshold**
- When decoder uncertainty > threshold → stop
- Simple but might not be differentiable

**Option B: Soft gating**
- Decoder outputs a "completion probability"
- Trained end-to-end
- More flexible

**Option C: External signal**
- Use byte-level-model's surprise signal
- When byte-level-model is surprised, decoder stops
- Couples the two signals

**Recommendation:** Start with Option B (soft gating), most flexible.

## Prove-One-Thing Experiment

**Claim:** Shared-state architecture (byte-level-state = patch, dynamic boundaries, decoder lookahead) outperforms separate-state architecture (separate byte/patch models with additive fusion).

**Experiment:**
1. Build shared-state architecture as described above
2. Train on same 99KB TinyStories slice, 1000 steps, cosine LR
3. Compare final loss against step 5 (RWKV byte + large transformer patch, loss 0.071)
4. If shared-state beats 0.071, the architecture matters (not just capacity)

**Success criteria:** Shared-state achieves lower loss than step 5 at matched params.

**Ablations (if shared-state wins):**
- Remove dynamic boundaries → use fixed patches → does performance drop?
- Remove decoder lookahead → decoder only gets current patch → does performance drop?
- Remove patch-model predictions → decoder only gets byte-level-state → does performance drop?

Each ablation isolates one component of the architecture.

## Relationship to Speculative Decoding

The shared-state architecture is similar to speculative decoding:
- Patch-model "drafts" future patches (like a small fast model drafts tokens)
- Decoder "verifies" by generating bytes (like a large slow model verifies)
- Decoder uses lookahead context from patch-model (like verifier uses drafted tokens)

**Difference:** In speculative decoding, there's rollback on mismatch. In our architecture, there's no rollback — the decoder just continues generating. The "verification" is implicit in the loss function.

**Advantage:** No rollback means simpler training and inference. The decoder learns to use lookahead context without needing explicit verification.
